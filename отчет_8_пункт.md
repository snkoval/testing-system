## 1. Что реализовано

### 1.1. Архитектура (`app/checker.py`)

Ядро системы проверки — модуль `app/checker.py`, предоставляющий функцию `check_submission(code, language, task_id)`, которая:

1. Загружает тесты задачи из файлов (`in_N.txt` / `out_N.txt`).
2. Читает лимиты времени и памяти из БД (`Task.time_limit`, `Task.memory_limit`).
3. Для C++ — компилирует код через `g++`.
4. Для каждого теста запускает программу, контролируя время и память.
5. Сравнивает вывод с эталоном (с нормализацией пробелов и переносов).
6. Возвращает структурированный результат.

### 1.2. Функция `_run_command` — единый запуск с лимитами

Универсальная функция для запуска как Python, так и скомпилированного C++:

| Параметр | Назначение |
|---|---|
| `cmd` | Команда запуска (`[python, script.py]` или `[exe.exe]`) |
| `test_input` | Входные данные теста (передаются через `stdin`) |
| `time_limit` | Лимит времени в секундах |
| `memory_limit` | Лимит памяти в КБ |

**Алгоритм работы:**

1. Процесс запускается через `subprocess.Popen` с пайпами для `stdin`/`stdout`/`stderr`.
2. Первоначальный `communicate(input=test_input, timeout=0.1)` — отправляет входные данные. Если процесс завершился за 0.1 с — результат возвращается немедленно.
3. Если процесс продолжает работать — enters polling loop:
   - Проверка лимита времени: `time.monotonic() - start > time_limit` → статус `time_limit`.
   - Проверка лимита памяти: `_tree_rss_kb(p) > memory_limit` → статус `memory_limit`.
   - `communicate(timeout=0.1)` — ожидание завершения с интервалом 0.1 с.
4. При превышении лимита — процесс принудительно завершается (`proc.kill()`).

### 1.3. Контроль памяти — `_tree_rss_kb` (раздел 7.3 ТЗ)

**Проблема:** На Windows `sys.executable` в виртуальном окружении (venv) является лаунчером, который запускает дочерний процесс реального интерпретатора. `psutil.Process(proc.pid)` видит только лаунчер (~3 МБ), а реальное потребление памяти — в дочернем процессе.

**Решение:** Функция `_tree_rss_kb(process)` суммирует RSS процесса **и всех его дочерних процессов** (рекурсивно через `process.children(recursive=True)`). Это корректно работает как с venv-лаунчером, так и с прямым вызовом интерпретатора.

### 1.4. Компиляция C++ — `_compile_cpp`

| Этап | Детали |
|---|---|
| Вызов компилятора | `g++ -o <exe> <source.cpp>` (путь из `CPP_COMPILER` конфигурации) |
| Таймаут компиляции | 10 секунд |
| Ошибки | `returncode != 0` → текст `stderr` возвращается как `compile_error` |
| Компилятор не найден | `FileNotFoundError` → сообщение «Компилятор не найден» |

### 1.5. Нормализация вывода — `_normalize_output`

Сравнение вывода программы с эталоном производится после нормализации:

1. Разбиение на строки.
2. Удаление trailing пробелов в каждой строке.
3. Удаление пустых строк в конце.

Это позволяет игнорировать различия в trailing whitespace и лишних переносах строк.

### 1.6. Запуск Python — `_run_python`

1. Код записывается во временный файл (`tempfile.NamedTemporaryFile`, суффикс `.py`).
2. Вызывается `_run_command([python_exec, source_path], ...)`.
3. Временный файл удаляется в блоке `finally`.

Путь к интерпретатору берётся из `PYTHON_EXEC` конфигурации.

### 1.7. Функция `check_submission` — основной API

```python
check_submission(code, language, task_id) → dict
```

**Возвращаемая структура:**

```python
{
    "status": "ok" | "error",
    "total_tests": N,
    "passed": M,
    "results": [
        {"number": 1, "status": "passed", "message": ""},
        {"number": 2, "status": "failed", "message": "Неверный ответ"},
        {"number": 3, "status": "time_limit", "message": "Превышено ограничение времени"},
        {"number": 4, "status": "memory_limit", "message": "Превышено ограничение памяти"},
        {"number": 5, "status": "runtime_error", "message": "Ошибка выполнения"},
    ],
    "compile_error": None | "текст ошибки"
}
```

**Возможные статусы теста:**

| Статус | Условие |
|---|---|
| `passed` | Вывод совпал с эталоном (после нормализации) |
| `failed` | Вывод не совпал — «Неверный ответ» |
| `time_limit` | Превышен лимит времени |
| `memory_limit` | Превышен лимит памяти |
| `runtime_error` | Ненулевой код возврата |

## 4. Технические решения

- **`Popen` вместо `subprocess.run`** — для контроля памяти необходимо опрашивать процесс во время выполнения. `subprocess.run` блокирует до завершения, не позволяя проверять память. Используется `Popen` с `communicate(timeout=0.1)` в цикле опроса.

- **Мониторинг дерева процессов (`_tree_rss_kb`)** — venv-лаунчер Windows запускает реальный интерпретатор как дочерний процесс. Память лаунчера (~3 МБ) не отражает реальное потребление. Функция суммирует RSS всех процессов в дереве.

- **Интервал опроса 0.1 с** — компромисс между точностью обнаружения и накладными расходами. Процесс проверяется каждые 100 мс на превышение времени и памяти.

- **`memory_limit=0` → без контроля** — если лимит равен нулю, проверка памяти пропускается (`if memory_limit:`).

- **Нормализация вывода** — сравнение производится после удаления trailing пробелов и пустых строк в конце. Это позволяет программам с различным форматированием (лишние пробелы, переносы) проходить тесты.

- **Первый `communicate` с `input`** — входные данные теста передаются при первом вызове `communicate(input=test_input, timeout=0.1)`. Последующие вызовы в цикле опроса не передают `input` (Python запрещает повторную отправку после начала коммуникации).

## 6. Исправленные ошибки

| Проблема | Решение |
|---|---|
| Контроль памяти отсутствовал — `memory_limit` хранился в БД и отображался в UI, но не enforced при выполнении кода | Реализован мониторинг RSS дерева процессов через `psutil` с интервалом 0.1 с |
| `psutil.Process(pid).memory_info()` показывал ~3 МБ для процесса с 100 МБ выделенной памяти | Обнаружено: venv-лаунчер Windows запускает реальный интерпретатор как дочерний процесс. Решение — `_tree_rss_kb()` суммирует RSS всех процессов в дереве |
| `subprocess.run()` блокирует до завершения, не позволяя проверять память во время выполнения | Переход на `subprocess.Popen` + `communicate(timeout=0.1)` в цикле опроса |
| Тестовые фикстуры использовали `memory_limit=256` (КБ) — слишком мало для Python (интерпретатор использует ~15 МБ) | Фикстуры обновлены до `memory_limit=262144` (256 МБ) |
