## 1. Что реализовано

### 1.1. Маршруты управления задачами (`app/teacher.py`)

| Маршрут | Метод | Описание |
|---|---|---|
| `/teacher/lessons/<id>/tasks` | GET | Список задач урока: буквенный индекс, краткое название, количество тестов, лимиты (время/память), действия (редактировать, тесты, удалить) |
| `/teacher/lessons/<id>/tasks/create` | GET, POST | Создание задачи: все поля условия, лимиты (по умолчанию 1 с / 256 КБ). `letter_index` присваивается автоматически (A, B, C, …) |
| `/teacher/tasks/<id>/edit` | GET, POST | Редактирование задачи: те же поля, что при создании; буквенный индекс не меняется |
| `/teacher/tasks/<id>/delete` | POST | Удаление задачи (каскадное удаление тестов и отправок) |
| `/teacher/tasks/<id>/tests` | GET, POST | Управление тестами: список с превью, добавление, редактирование, удаление |

**Логика автоматического присвоения буквенного индекса:**
- Если задач в уроке нет — первой присваивается буква `A`.
- Если задачи есть — следующая буква вычисляется как `chr(ord(max(существующие_буквы)) + 1)`.
- Пример: при наличии `A`, `B` новая задача получает `C`.

**Логика создания задачи:**
- Проверка лимита: **не более 20 задач на урок** (раздел 6.2 ТЗ) — при превышении возвращается форма с ошибкой, задача не создаётся.
- Обязательные поля: краткое название, текст условия, описание входных данных, описание результата. При отсутствии — возврат формы с `flash`-сообщением.
- Необязательные поля: пояснения (`notes`).
- Лимиты времени/памяти: если поле пустое или не число — применяются значения по умолчанию (1 с / 256 КБ); иначе — введённые значения.

### 1.2. Управление тестами (`app/teacher.py` + `app/test_files.py`)

Тесты хранятся в файлах (реализовано в подзадаче 2), а не в БД — раздел 7.2 ТЗ.

Структура хранения:
```
tests_files/
└── <task_id>/
    ├── in_1.txt    out_1.txt
    ├── in_2.txt    out_2.txt
    └── ...
```

Действия на странице `/teacher/tasks/<id>/tests` (единый маршрут, параметр `action`):

| Действие | Параметр `action` | Поведение |
|---|---|---|
| Добавить тест | `add` | Новый номер = `max(номера) + 1`; создаёт файлы `in_N.txt` / `out_N.txt` |
| Редактировать тест | `edit` | Перезаписывает содержимое `in_N.txt` / `out_N.txt` |
| Удалить тест | `delete` | Удаляет файлы теста; **блокируется, если останется меньше 2 тестов** |

**Защита минимума тестов:** согласно ТЗ (раздел 7.2) минимальное количество тестов — 2 (тесты 1 и 2 нужны для автогенерации примеров в условии). При попытке удалить тест, когда в наборе всего 2, выводится ошибка, тест сохраняется.

**Редактирование доступно и после публикации урока** — ограничений по статусу урока нет (раздел 7.2 ТЗ).

### 1.3. Шаблоны

Созданы 3 шаблона:

| Шаблон | Назначение |
|---|---|
| `teacher/tasks.html` | Таблица задач урока с действиями. Для каждой задачи показывается количество тестов (`get_test_count`) и лимиты. Удаление — через форму с подтверждением |
| `teacher/task_form.html` | Единая форма создания/редактирования (переключается по наличию `task` в контексте). Поля: краткое название, текст условия, описание ввода, описание результата, пояснения, лимит времени, лимит памяти |
| `teacher/tests.html` | Таблица существующих тестов с превью (до 100 символов) + раскрывающиеся формы редактирования каждого теста (`<details>`) + форма добавления нового теста |

Все шаблоны наследуются от `base.html`, используют `get_flashed_messages()` и `csrf_token()`.

## 4. Технические решения

- **Автонумерация буквенных индексов** через `chr(ord(max(...)) + 1)` — корректно продолжает последовательность `A, B, C, …` независимо от того, удалялись ли промежуточные задачи (берётся максимум из существующих).
- **Лимит 20 задач** проверяется до валидации полей (`len(existing) >= 20`) — даже при невалидных данных 21-я задача не создаётся.
- **Единый маршрут `/tests`** обрабатывает все три действия (add/edit/delete) через скрытое поле `action` — упрощает маршрутизацию и шаблоны.
- **Минимум 2 теста** защищён на уровне маршрута: `get_test_count(task_id) <= 2` → отказ с `flash`-сообщением.
- **Нумерация тестов сквозная** — новый номер всегда `max(существующие) + 1`, поэтому удаление среднего теста оставляет «дыру» в нумерации (например, после удаления теста 2 остаются 1 и 3). Это соответствует файловой модели `in_N.txt` и не влияет на проверку.
- **Превью в таблице тестов** ограничено 100 символами с многоточием — защита интерфейса от переполнения длинными тестами.
- **Редактирование теста через `<details>`** — раскрывающаяся форма для каждого теста, не загромождает страницу.
- **Каскадное удаление** задачи настроено в модели `Task` (`cascade='all, delete-orphan'`): удаляются связанные `Submission`; файлы тестов остаются на диске, но становятся недоступны (нет task_id).
