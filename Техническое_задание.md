# Техническое задание

## Проект: «Система тестирования программ учеников»

---

## 1. Назначение и цели

### 1.1. Назначение

Веб-приложение предназначено для проверки и автоматического тестирования программ, написанных учениками в рамках учебных занятий. Система поддерживает два режима работы: **режим учителя** (администратор) и **режим ученика**.

### 1.2. Цели

- Автоматизировать процесс проверки программного кода, написанного учениками.
- Обеспечить учителю удобное управление группами, аккаунтами, уроками и задачами.
- Обеспечить ученикам простой и безопасный доступ к учебным материалам и тестированию программ.
- Гарантировать конфиденциальность персональных данных учеников.

## 3. Режим учителя

### 3.1. Аутентификация учителя

- Так как приложение развёртывается на **локальном сервере** и учителя только **один**, регистрация через веб-интерфейс не предусмотрена.
- Вход в режим учителя осуществляется по **логину и паролю**, которые **задаются при первичной настройке сервера** (в конфигурационном файле или скриптом инициализации).
- Страница аутентификации учителя доступна по отдельному маршруту — `/admin` (прямая ссылка).
- После успешной аутентификации учитель получает доступ к панели администрирования.
- В панели администрирования предусмотрена возможность **смены пароля** учителя.

### 3.2. Управление группами учеников

#### 3.2.1. Создание группы

- Учитель создаёт группы учеников.
- **Название группы** формируется по маске:

  ```
  <класс>_<группа>gr
  ```

  где:
  - `<класс>` — номер класса (одна или две цифры) и литера класса (одна заглавная буква), например `7A`, `10B`;
  - `_` — разделитель (символ подчёркивания);
  - `<группа>` — номер группы (одна цифра), задаётся учителем;
  - `gr` — литералы «gr» (от слова «группа»).

  **Пример:** `7A_1gr`, `10B_2gr`.

#### 3.2.2. Управление аккаунтами учеников

- Внутри каждой группы учитель создаёт аккаунты учеников.
- Предусмотрены **два режима создания**:
  - **Массовое создание** — учитель указывает количество учеников и вводит их фамилии и имена списком; система автоматически генерирует логины и пароли для всех;
  - **Добавление по одному** — учитель добавляет отдельного ученика при необходимости.
- Для каждого ученика фиксируются:
  - **Фамилия**;
  - **Имя**;
  - **Группа** (группа, к которой привязан ученик).

- **Данные ученика (фамилия, имя, группа) видны только учителю** и недоступны другим ученикам.

- Вход в систему для учеников выполняется по **логину и паролю**, без дополнительной аутентификации.

#### 3.2.3. Формирование логина ученика

- Логин формируется по шаблону:

  ```
  <маска_группы>_<порядковый_номер>
  ```

  где:
  - `<маска_группы>` — название группы, к которой принадлежит ученик;
  - `<порядковый_номер>` — порядковый номер ученика в группе (начиная с 1).

  **Пример:** для группы `7A_1gr` первый ученик получает логин `7A_1gr_1`, второй — `7A_1gr_2` и т.д.

#### 3.2.4. Формирование пароля ученика

- Пароль — **произвольный набор символов**.
- Допустимые символы:
  - Строчные латинские буквы (`a–z`);
  - Цифры (`0–9`);
  - Специальные символы.
- Заглавные буквы (Caps Lock) **не используются**.
- Максимальная длина пароля — **6 символов**.
- Учитель имеет возможность:
  - **Сменить пароль** ученика вручную;
  - **Сбросить пароль** (система генерирует новый случайный пароль по заданным правилам).

## 4. Доступ ученика к системе

### 4.1. Страница аутентификации ученика

- **Вход на страницу аутентификации возможен только по прямой ссылке.**
- Прямая ссылка сообщается ученикам учителем.
- На странице вводятся логин и пароль, выданные учителем.
- Дополнительная аутентификация (подтверждение по e-mail, SMS и т.п.) **не требуется**.

### 4.2. Безопасность и конфиденциальность

- Персональные данные учеников (фамилия, имя, группа) **не отображаются** в интерфейсе ученического режима и **недоступны** другим ученикам.
- Ученик видит только учебные материалы и результаты собственных программ.

## 5. Уроки

### 5.1. Создание уроков

- Уроки создаёт учитель.
- Каждый урок характеризуется:
  - **Порядковый номер** (целое число);
  - **Тема урока** (строка);
  - **Ссылка на теоретический материал** (необязательное поле).

### 5.2. Страница списка уроков (для учеников)

- После аутентификации ученик видит **страницу со списком уроков**.
- Каждый урок в списке представлен как **ссылка**.
- **Доступ к урокам открывает учитель.** Пока учитель не открыл урок, он **не отображается** в списке учеников своей группы.
- **Новые ссылки на уроки располагаются выше старых** (сортировка по убыванию порядкового номера / даты создания).

### 5.3. Управление доступом к урокам

- Учитель имеет возможность открывать и закрывать доступ ученикам к конкретным урокам.
- При открытии урока учитель задаёт **срок доступа**:
  - **На определённое количество дней** (после истечения срока урок автоматически скрывается из списка учеников);
  - **Бессрочно** (доступ остаётся открытым до явного закрытия учителем).
- Состояние доступа (открыт / закрыт, срок действия) управляется учителем и хранится в базе данных.

## 6. Страница урока

### 6.1. Структура страницы

При переходе к уроку открывается страница, содержащая **три фрейма**:

```
┌──────────────────────────────────────────────────────┐
│  Фрейм 1: Список задач урока (левая колонка)          │
│  ─────────────────────────────────────────────────    │
│  A. Краткое название задачи 1                         │
│  B. Краткое название задачи 2                         │
│  C. Краткое название задачи 3                         │
│  ...                                                  │
├──────────────────────────────────────────────────────┤
│  Фрейм 2: Текст условия задачи (верхняя часть)        │
│  ─────────────────────────────────────────────────    │
│  Полное текстовое описание условия выбранной задачи   │
├──────────────────────────────────────────────────────┤
│  Фрейм 3: Окно ввода кода и вывода результатов        │
│  ─────────────────────────────────────────────────    │
│  Текстовое поле для вставки кода программы             │
│  Кнопка «Проверить»                                    │
│  Область вывода сообщений о результатах тестирования   │
└──────────────────────────────────────────────────────┘
```

### 6.2. Фрейм 1 — Список задач урока

- Располагается **в левой колонке** страницы.
- Содержит **нумерованный список задач** урока.
- **Нумерация задач — латинскими буквами** (`A`, `B`, `C`, `D`, …).
- Для каждой задачи отображается **краткое название**.
- Количество задач в одном уроке: **от 1 до 20**.
- При выборе задачи из списка её условие и область ввода кода загружаются в соответствующие фреймы.

### 6.3. Фрейм 2 — Текст условия задачи

- Располагается в **верхней части основной области** окна.
- Содержит **полный текст условия** выбранной задачи.
- **Полный текст условия** состоит из 6 разделов: в первом разделе указывается ограничение по времени работы программы, ограничения по объёму используемой памяти; во втором разделе — текст условия задачи; в третьем — описание вводимых данных, в четвёртом — описание результата; в пятом (в виде таблицы из двух колонок) — пример данных и результата (вставляются автоматически из 1 и 2 теста, используемых для проверки работы кода); в шестом — пояснения (необязательный раздел).

### 6.4. Фрейм 3 — Окно ввода кода и результаты тестирования

- Располагается в **нижней части основной области** окна.
- Содержит:
  - **Текстовое поле (редактор)** для вставки/написания кода программы с **подсветкой синтаксиса** выбранного языка (Python или C++);
  - **Выбор языка программирования** (Python или C++ (GNU GCC Compiler));
  - **Кнопку отправки кода на проверку** (например, «Проверить» / «Отправить»);
  - **Область вывода сообщений о результатах тестирования** — отображает:
    - Статус проверки (успешно / ошибка);
    - Результаты по каждому тесту (пройден / не пройден);
    - Текст ошибок компиляции или выполнения (при наличии);
    - Итоговое сообщение о корректности программы.

## 7. Требования к тестированию программ

### 7.1. Процесс проверки

1. Ученик выбирает задачу из списка (Фрейм 1).
2. Знакомится с условием задачи (Фрейм 2).
3. Вставляет или пишет код программы в редакторе (Фрейм 3).
4. Нажимает кнопку отправки на проверку.
5. Сервер выполняет код ученика на наборе тестов, заранее заданных учителем для данной задачи.
6. Результаты тестирования выводятся в области вывода (Фрейм 3).

### 7.2. Состав набора тестов

- Для каждой задачи учитель задаёт набор тестов в виде **группы файлов**:
  - `in_<номер_теста>.txt` — файл с входными данными;
  - `out_<номер_теста>.txt` — файл с ожидаемым результатом.
  - **Пример:** `in_1.txt` / `out_1.txt`, `in_2.txt` / `out_2.txt` и т.д.
- Минимальное количество тестов для одной задачи — **2** (тесты 1 и 2 используются для автоматического формирования примеров в условии).
- Количество тестов для одной задачи определяется учителем.
- При проверке программа ученика запускается на каждом тесте.
- **Редактирование тестов после публикации урока обязательно** — учитель может добавлять, изменять и удалять тесты в любой момент.

### 7.3. Ограничения ресурсов

- Для каждой задачи учитель настраивает **ограничения выполнения**:
  - **Лимит времени** — максимальное время выполнения программы на одном тесте;
  - **Лимит памяти** — максимальный объём памяти, доступный программе.
- **Значения по умолчанию:**
  - Время — **1 секунда**;
  - Память — **256 КБ**.
- При превышении лимита тест считается не пройденным с соответствующим сообщением («Превышено ограничение времени» / «Превышено ограничение памяти»).
- Языки программирования, поддерживаемые системой: **Python** (версия поддерживает обновление) и **C++** (GNU GCC Compiler). Ученик выбирает язык перед отправкой кода.

### 7.4. Отображение результатов

Для каждой задачи ученику отображается:

| Элемент | Описание |
|---|---|
| Общее число тестов | Количество тестов в наборе |
| Число пройденных тестов | Сколько тестов выполнено успешно |
| Статус по каждому тесту | Пройден / Не пройден |
| Сообщения об ошибках | Текст ошибок компиляции или времени выполнения |
| Итоговый статус | Задача решена полностью / частично / не решена |

## 9. Требования к интерфейсу

### 9.1. Общие требования

- Интерфейс должен быть **понятным и простым**, ориентированным на школьников, без лишних элементов дизайна, синий или белый фон, контрастный шрифт текста, рамки.
- Поддержка работы в **современных браузерах** (Chrome, Firefox, Edge, Safari).
- Адаптивная вёрстка для возможности работы на устройствах с разным размером экрана (настольные ПК, ноутбуки, планшеты).

### 9.2. Интерфейс учителя

- Панель администрирования с разделами:
  - **Группы** — создание, редактирование, удаление групп.
  - **Ученики** — создание, редактирование, удаление аккаунтов; генерация логинов и паролей.
  - **Уроки** — создание, редактирование, управление доступом (срок доступа в днях или бессрочно).
  - **Задачи** — создание задач, ввод условий, настройка тестов, лимитов времени и памяти.
  - **Результаты** — сводная таблица результатов всех учеников по задачам и урокам с возможностью **экспорта в Excel**.

### 9.3. Интерфейс ученика

- **Страница аутентификации** (доступ по прямой ссылке) — поля «Логин» и «Пароль», кнопка «Войти».
- **Страница списка уроков** — список ссылок на доступные уроки (новые сверху).
- **Страница урока** — три фрейма в соответствии с п. 6.

## 10. Таблица результатов и экспорт

### 10.1. Сводная таблица результатов

- В режиме учителя формируется **сводная таблица результатов** всех учеников.
- Таблица формируется отдельно по каждой группе, отображает по каждому ученику и каждой задаче:
  - Количество успешно пройденных тестов или признак полного решения (все тесты пройдены - 'OK');
  - если ученик не приступал к решению задачи, то в  ячейка соответствующая этой задачи пуста.  
- Уведомления учителю **не отправляются** — вся информация доступна в таблице по запросу.

### 10.2. Экспорт в Excel

- Сводная таблица результатов может быть **экспортирована в формат Excel** (`.xlsx`). 
- Экспорт доступен из раздела «Результаты» панели учителя.

## 11. Требования к безопасности

1. Пароли хранятся в базе данных **только в хэшированном виде**.
2. Персональные данные учеников **не передаются** другим ученикам ни в каком виде.
3. Доступ к административным функциям (режим учителя) защищён аутентификацией.
4. Страница входа ученика **не индексируется** и доступна только по прямой ссылке.
5. Предусмотреть защиту от типичных веб-уязвимостей:
   - SQL-инъекции (параметризованные запросы);
   - XSS (экранирование вывода);
   - CSRF (токены для форм).
6. Выполнение кода учеников должно происходить в **изолированной среде (песочнице)** для предотвращения несанкционированного доступа к серверу.

## 12. Требования к развёртыванию

1. Приложение может быть развёрнуто на **разных серверах** независимо.
2. Каждый экземпляр сервера настраивается на **одного учителя** со своим набором данных.
3. Процесс установки должен включать:
   - Размещение серверной части на целевом сервере;
   - Настройку подключения к базе данных;
   - Создание учётной записи учителя при первичной настройке;
   - Установку среды выполнения для проверки кода учеников (песочница).
