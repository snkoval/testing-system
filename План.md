# План реализации проекта «Система тестирования программ учеников»

---

## Обзор

Проект реализуется на стеке: **Python (Flask)** + **SQLite (SQLAlchemy)** + **HTML/CSS/JavaScript**. План разбит на 12 подзадач, каждая из которых может разрабатываться и тестироваться независимо. Между подзадачами указаны зависимости.

> **Методология:** Реализация каждой подзадачи должна быть построена по принципам **TDD (Test-Driven Development)** — сначала пишутся тесты (красная фаза), затем пишется минимальный код для их прохождения (зелёная фаза), затем выполняется рефакторинг. Для тестирования используется `pytest` + `pytest-flask`.

## Подзадача 1. Подготовка проекта и инфраструктуры

**Цель:** создать структуру каталогов, настроить виртуальное окружение, базовый Flask-проект.

### Шаги

1. **Создать структуру каталогов:**

   ```
   testing-system/
   ├── app/
   │   ├── __init__.py            # фабрика приложения (create_app)
   │   ├── config.py              # конфигурация (SECRET_KEY, БД, лимиты)
   │   ├── models.py              # модели SQLAlchemy
   │   ├── auth.py                # blueprint аутентификации
   │   ├── teacher.py             # blueprint панели учителя
   │   ├── student.py             # blueprint интерфейса ученика
   │   ├── checker.py             # модуль тестирования кода
   │   ├── templates/
   │   │   ├── base.html          # базовый шаблон
   │   │   ├── auth/
   │   │   ├── teacher/
   │   │   └── student/
   │   ├── static/
   │   │   ├── css/
   │   │   ├── js/
   │   │   └── vendor/            # CodeMirror / Ace Editor
   │   └── uploads/               # временные файлы кода учеников
   ├── tests/                     # файлы тестов проекта
   ├── instance/                  # конфигурация развёртывания, БД
   ├── requirements.txt
   ├── run.py                     # точка входа
   └── init_db.py                 # скрипт инициализации БД + учителя
   ```

2. **Создать виртуальное окружение** `python -m venv venv`.

3. **Создать `requirements.txt`** с зависимостями:

   ```
   flask
   flask-sqlalchemy
   werkzeug.security (входит в Werkzeug)
   openpyxl          # экспорт в Excel
   ```

4. **Создать `app/config.py`** — класс конфигурации:

   - `SECRET_KEY`
   - `SQLALCHEMY_DATABASE_URI = 'sqlite:///testing.db'`
   - Дефолтные лимиты: `DEFAULT_TIME_LIMIT = 1`, `DEFAULT_MEMORY_LIMIT = 256`
   - Пути к компиляторам: `PYTHON_EXEC`, `CPP_COMPILER`

5. **Создать фабрику приложения** `app/__init__.py` (функция `create_app()`):
   - Инициализация SQLAlchemy
   - Регистрация blueprints
   - Создание БД при первом запуске

6. **Создать `run.py`** — запуск `create_app().run()`.

7. **Проверить:** `python run.py` запускает сервер, главная страница возвращает базовый шаблон.

## Подзадача 2. Проектирование базы данных

**Цель:** описать все модели данных в SQLAlchemy, соответствующие разделу 8 ТЗ.

### Шаги

1. **Модель `Teacher`** (раздел 8.1.1):

   ```python
   id, username, password_hash
   ```

2. **Модель `Group`** (раздел 8.1.2):

   ```python
   id, name  # маска: <класс>_<группа>gr
   ```

   - Связь `students` (one-to-many с `Student`).

3. **Модель `Student`** (раздел 8.1.3):

   ```python
   id, group_id, login, password_hash, last_name, first_name, seq_number
   ```

   - `login` — уникальный, генерируется автоматически.
   - `password_hash` — хэш пароля (до 6 символов в открытом виде).

4. **Модель `Lesson`** (раздел 8.1.4):

   ```python
   id, order_number, title, theory_url, is_open, access_days (nullable), opened_at (nullable)
   ```

   - `is_open` — флаг (да/нет).
   - `access_days` — целое (количество дней) или `NULL` (бессрочно).
   - `opened_at` — дата/время открытия (для расчёта истечения срока).

5. **Модель `Task`** (раздел 8.1.5):

   ```python
   id, lesson_id, letter_index, short_title, problem_text,
   input_description, output_description, notes (nullable),
   time_limit (default 1), memory_limit (default 256)
   ```

   - `letter_index` — латинская буква `A`–`T` (максимум 20 задач на урок).
   - Тесты хранятся в файлах (см. шаг 6).

6. **Хранение тестов (файлы, не БД)** — раздел 7.2:

   - Каталог: `tests/<task_id>/`
   - Файлы: `in_1.txt`, `out_1.txt`, `in_2.txt`, `out_2.txt`, …
   - Создать вспомогательные функции в `models.py` или отдельном модуле `test_files.py`:
     - `get_test_dir(task_id) → Path`
     - `get_tests(task_id) → list[(number, input, expected_output)]`
     - `add_test(task_id, number, input, output)`
     - `update_test(task_id, number, input, output)`
     - `delete_test(task_id, number)`
     - `get_examples(task_id) → [(in_1, out_1), (in_2, out_2)]` — для автогенерации раздела 5 условия.

7. **Модель `Submission`** (раздел 8.1.6):

   ```python
   id, student_id, task_id, code, language, test_results (JSON),
   submitted_at
   ```

   - `code` — только последняя версия (перезаписывается при повторной отправке).
   - `language` — `'python'` или `'cpp'`.
   - `test_results` — JSON со статусом каждого теста.
   - Уникальное ограничение: `(student_id, task_id)` — одна запись на пару «ученик + задача».

8. **Создать `init_db.py`**:
   - Создание всех таблиц.
   - Создание единственной записи `Teacher` с логином и паролем из конфигурации.

9. **Проверить:** `python init_db.py` создаёт БД, таблицы видны в SQLite Browser.

## Подзадача 3. Аутентификация и авторизация

**Цель:** реализовать вход учителя (`/admin`) и ученика (прямая ссылка), управление сессиями.

### Шаги

1. **Создать `app/auth.py`** (blueprint `auth`).

2. **Вход учителя — `/admin`** (раздел 3.1):
   - GET — форма входа (логин + пароль).
   - POST — проверка `Teacher` в БД, `check_password_hash()`.
   - При успехе — запись в `session['teacher_id']`.
   - Редирект на панель учителя `/teacher`.

3. **Вход ученика — `/login`** (раздел 4.1):
   - GET — форма входа (логин + пароль).
   - POST — проверка `Student` в БД.
   - При успехе — запись в `session['student_id']`.
   - Редирект на список уроков `/lessons`.

4. **Декораторы доступа:**
   - `@login_required_teacher` — проверяет `session['teacher_id']`, иначе редирект на `/admin`.
   - `@login_required_student` — проверяет `session['student_id']`, иначе редирект на `/login`.

5. **Выход:**
   - `GET /logout` — очищает сессию, редирект на соответствующую страницу входа.

6. **Смена пароля учителя** (раздел 3.1):
   - `GET/POST /teacher/settings` — форма смены пароля (старый пароль, новый, подтверждение).

7. **Шаблоны:**
   - `auth/teacher_login.html`
   - `auth/student_login.html`

8. **Проверить:**
   - Вход учителя с правильным/неправильным паролем.
   - Вход ученика с правильным/неправильным паролем.
   - Доступ к защищённым страницам без логина → редирект.

## Подзадача 4. Управление группами и учениками

**Цель:** панель учителя для создания групп и управления аккаунтами учеников.

### Шаги

1. **Создать `app/teacher.py`** (blueprint `teacher`, префикс `/teacher`).

2. **Список групп — `GET /teacher/groups`**:
   - Таблица: название группы, количество учеников, действия (просмотр, удаление).

3. **Создание группы — `GET/POST /teacher/groups/create`** (раздел 3.2.1):
   - Форма: номер класса (1–2 цифры), литера класса (заглавная буква), номер группы (цифра).
   - Валидация маски `<класс>_<группа>gr`.
   - Уникальность названия.

4. **Просмотр группы — `GET /teacher/groups/<id>`**:
   - Таблица учеников: логин, пароль (в открытом виде для выдачи), фамилия, имя, действия.
   - Кнопки: «Добавить ученика», «Массовое создание».

5. **Добавление одного ученика — `POST /teacher/groups/<id>/add`** (раздел 3.2.2):
   - Форма: фамилия, имя.
   - `seq_number` — следующий порядковый номер в группе.
   - Логин генерируется: `<название_группы>_<seq_number>`.
   - Пароль генерируется: 6 символов (строчные буквы, цифры, спецсимволы).

6. **Массовое создание — `POST /teacher/groups/<id>/bulk-add`** (раздел 3.2.2):
   - Форма: текстовое поле, по строке на ученика (формат: `Фамилия Имя`).
   - Парсинг строк, автоматическая генерация логинов и паролей.
   - Отображение результата: список созданных аккаунтов с логинами и паролями.

7. **Управление паролями** (раздел 3.2.4):
   - **Смена пароля** — `POST /teacher/students/<id>/password` (ручной ввод нового пароля).
   - **Сброс пароля** — `POST /teacher/students/<id>/reset-password` (автогенерация нового).

8. **Функции-утилиты:**
   - `generate_password()` — генерация случайного пароля по правилам ТЗ.
   - `generate_login(group_name, seq_number)` — формирование логина.

9. **Шаблоны:**
   - `teacher/groups.html`
   - `teacher/group_detail.html`
   - `teacher/student_form.html`
   - `teacher/bulk_add.html`

10. **Проверить:**
    - Создание группы с валидной/невалидной маской.
    - Массовое создание учеников, проверка логинов и паролей.
    - Добавление одного ученика.
    - Смена и сброс пароля.

## Подзадача 5. Управление уроками и доступом

**Цель:** создание, редактирование уроков и управление доступом учеников.

### Шаги

1. **Список уроков (учитель) — `GET /teacher/lessons`**:
   - Таблица: порядковый номер, тема, статус (открыт/закрыт), срок доступа, действия.

2. **Создание урока — `GET/POST /teacher/lessons/create`** (раздел 5.1):
   - Форма: порядковый номер, тема, ссылка на теорию (необязательно).
   - Урок создаётся в статусе «закрыт».

3. **Редактирование урока — `GET/POST /teacher/lessons/<id>/edit`**:
   - Изменение порядкового номера, темы, ссылки.

4. **Управление доступом — `POST /teacher/lessons/<id>/access`** (раздел 5.3):
   - Открыть доступ:
     - Поле «срок в днях» (целое) или переключатель «бессрочно».
     - Запись `is_open=True`, `opened_at=now()`, `access_days`.
   - Закрыть доступ: `is_open=False`.

5. **Проверка истечения срока** — функция `is_lesson_accessible(lesson)`:
   - Если `access_days is None` → доступен бессрочно.
   - Иначе: `now() <= opened_at + timedelta(days=access_days)`.

6. **Удаление урока — `POST /teacher/lessons/<id>/delete`**:
   - Каскадное удаление задач, тестов, отправок.

7. **Шаблоны:**
   - `teacher/lessons.html`
   - `teacher/lesson_form.html`

8. **Проверить:**
    - Создание урока, проверка дефолтного статуса «закрыт».
    - Открытие с лимитом дней и бессрочно.
    - Проверка функции `is_lesson_accessible` на истечение срока.

**Цель:** создание задач в рамках урока, загрузка и редактирование тестов.

### Шаги

1. **Список задач урока — `GET /teacher/lessons/<id>/tasks`**:
   - Таблица: буквенный индекс, краткое название, количество тестов, действия.

2. **Создание задачи — `GET/POST /teacher/lessons/<id>/tasks/create`** (раздел 8.1.5):
   - Поля:
     - Краткое название;
     - Текст условия;
     - Описание входных данных;
     - Описание результата;
     - Пояснения (необязательно);
     - Лимит времени (по умолчанию 1 с);
     - Лимит памяти (по умолчанию 256 КБ).
   - `letter_index` — присваивается автоматически (A, B, C, …) по порядку.

3. **Редактирование задачи — `GET/POST /teacher/tasks/<id>/edit`**:
   - Те же поля, что при создании.

4. **Управление тестами — `GET/POST /teacher/tasks/<id>/tests`** (раздел 7.2):

   - Редактирование теста.
   - Удаление теста.
   - Валидация: минимум 2 теста (тесты 1 и 2 нужны для примеров в условии).
   - **Редактирование тестов доступно и после публикации урока.**

5. **Шаблоны:**
   - `teacher/tasks.html`
   - `teacher/task_form.html`
   - `teacher/tests.html`

6. **Проверить:**
    - Создание задачи с лимитами по умолчанию и кастомными.

   - Синий или белый фон, контрастный шрифт, рамки (раздел 9.1).

## Подзадача 8. Система тестирования кода (песочница)

**Цель:** безопасное выполнение кода учеников на тестах с контролем времени и памяти.

### Шаги

1. **Создать `app/checker.py`** — ядро системы проверки.

2. **Сохранение кода** (раздел 8.1.6):
   - Код сохраняется во временный файл: `uploads/<submission_id>.py` или `.cpp`.

3. **Компиляция** (для C++):
   - Вызов `g++ -o uploads/<id> uploads/<id>.cpp` через `subprocess`.
   - Перехват ошибок компиляции → возврат сообщения.

4. **Выполнение на тестах** — функция `run_test(executable, test_input, time_limit, memory_limit)`:
   - Запуск через `subprocess.run()` с `timeout=time_limit`.
   - Передача `test_input` через `stdin`.
   - Чтение `stdout`.
   - Сравнение вывода с `out_N.txt` (с нормализацией пробелов/переносов).

5. **Контроль памяти** (раздел 7.3):
   - На Windows — через `psutil` после завершения процесса (пиковое использование).
   - На Linux — `resource.setrlimit(RLIMIT_AS, ...)` перед выполнением (при развёртывании на Linux).
   - При превышении — статус «Превышено ограничение памяти».

6. **Контроль времени** (раздел 7.3):
   - `subprocess.run(timeout=time_limit)`.
   - При `TimeoutExpired` — статус «Превышено ограничение времени».

7. **Изоляция (песочница)** (раздел 11.6):
   - Запуск во временной рабочей директории без доступа к другим файлам.
   - На Linux: опционально Docker-контейнер или `firejail`.
   - На Windows: ограниченный пользователь или контейнер.

8. **Функция `check_submission(code, language, task_id)` → dict:**
   ```python
   {
       "status": "ok" | "error",
       "total_tests": N,
       "passed": M,
       "results": [
           {"number": 1, "status": "passed", "message": ""},
           {"number": 2, "status": "failed", "message": "Wrong answer"},
           ...
       ],
       "compile_error": None | "текст ошибки"
   }
   ```

9. **Проверить (юнит-тесты):**
   - Правильный Python-код → все тесты пройдены.
   - Правильный C++-код → все тесты пройдены.
   - Код с бесконечным циклом → «Превышено ограничение времени».
   - Код с ошибкой компиляции C++ → сообщение об ошибке.
   - Код с неверным ответом → «Неверный ответ».
