## 1. Что реализовано

### 1.1. Хэширование паролей (раздел 11.1 ТЗ)

| Роль | Хранение пароля |
|---|---|
| Учитель | `Teacher.password_hash` — `generate_password_hash()` (Werkzeug) |
| Ученик | `Student.password_hash` — `generate_password_hash()`; `Student.password_plain` — для выдачи учителю в открытом виде при создании |

Проверка пароля — через `check_password_hash()`. Пароли никогда не хранятся в открытом виде в БД (поле `password_plain` используется только для отображения учителю при управлении аккаунтами).

**Тесты:** `TestPasswordHashing` — 2 теста (учитель, ученик).

### 1.2. Изоляция данных учеников (раздел 11.2 ТЗ)

| Защита | Реализация |
|---|---|
| Ученик видит только свои отправки | `Submission.query.filter_by(student_id=session['student_id'], ...)` — везде используется ID из сессии, а не из запроса |
| Ученик не может видеть код других | AJAX `/lessons/<id>/tasks/<id>` возвращает только `Submission` текущего ученика |
| Отправка кода привязана к сессии | `/submit` создаёт/обновляет `Submission` с `student_id=session['student_id']` |

**Тесты:** `TestDataIsolation` — 4 теста (не видит чужой код, отправка привязана к сессии, нет доступа к панели учителя, требуется вход).

### 1.5. Защита от веб-уязвимостей (раздел 11.5 ТЗ)

#### 1.5.1. SQL-инъекции

Все запросы к БД выполняются через **SQLAlchemy ORM** (параметризованные запросы):

- `Model.query.filter_by(field=value)` — параметризованный поиск
- `db.session.get(Model, id)` — получение по первичному ключу
- Никаких raw SQL-запросов с интерполяцией строк

#### 1.5.2. XSS (Cross-Site Scripting)

**Jinja2 автосклейка** включена по умолчанию во Flask. Все переменные в шаблонах автоматически экранируются:

- `{{ task.problem_text }}` → HTML-сущности (`<script>` → `&lt;script&gt;`)
- `{{ student.last_name }}` — экранирование

**Тесты:** `TestXSSProtection` — 1 тест (`<script>alert(1)</script>` в названии задачи экранируется).

#### 1.5.3. CSRF (Cross-Site Request Forgery)

**Реализация** (`app/utils.py`):

| Функция | Назначение |
|---|---|
| `generate_csrf_token()` | Генерация токена `secrets.token_hex(16)`, сохранение в `session['csrf_token']` |
| `validate_csrf_token()` | Сравнение токена из формы с токеном из сессии (`secrets.compare_digest` — защита от timing-атак). При несовпадении → `abort(400)` |

**Активация** (`app/__init__.py`):

```python
if app.config.get('CSRF_PROTECTION_ENABLED', False):
    @app.before_request
    def _check_csrf():
        if request.method in ('POST', 'PUT', 'DELETE'):
            validate_csrf_token()
```

CSRF-защита включена в production-конфигурации (`CSRF_PROTECTION_ENABLED = True`) и отключена в `TestConfig` (для удобства тестирования, кроме специализированных CSRF-тестов).

**Токены во всех формах:** 18 скрытых полей `<input type="hidden" name="csrf_token">` во всех POST-формах (teacher + student + auth). AJAX-отправка (`/submit`) добавляет `csrf_token` через `FormData`.

**Тесты:** `TestCSRFProtection` — 6 тестов (POST без токена → 400, с валидным → 302, с неверным → 400, создание группы без токена → 400, с токеном → 302, GET не блокируется).

## 4. Технические решения

- **CSRF через `before_request` вместо middleware** — глобальный хук проверяет все POST/PUT/DELETE запросы. Включается флагом `CSRF_PROTECTION_ENABLED` — позволяет отключать CSRF в unit-тестах (где нет рендеринга форм) и включать в production. Специальная `CSRFTestConfig` используется для тестирования самой CSRF-защиты.

- **`secrets.compare_digest()` для сравнения токенов** — защита от timing-атак. Обычное `==` может утечь информацию о токене через время сравнения.

- **`password_plain` в модели Student** — пароль хранится в открытом виде только для отображения учителю при создании аккаунта (для выдачи ученику). В перспективе можно зашифровать или удалить после первой выдачи. Основная аутентификация идёт через `password_hash`.

- **`@login_required_*` декораторы вместо `current_user`** — простая проверка `session['teacher_id']` / `session['student_id']`. Нет тяжёлой зависимости на Flask-Login — достаточно для системы с двумя ролями.

- **CSRF-токен в AJAX** — JavaScript в `lesson.html` добавляет `csrf_token` через `formData.append('csrf_token', '{{ csrf_token() }}')`. Токен рендерится в шаблон при загрузке страницы.
